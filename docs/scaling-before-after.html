Scaling the audit write path: V0 wrote each of the roughly 10 million audits per run synchronously to a standalone MongoDB, blocking the Kafka thread about 111 milliseconds per record for majority replication and topping out near 9 audits per second. The redesign forwards audits to a Kafka topic so the processing thread never waits on the database; a separate consumer drains 500 records per poll and writes them in one bulk write, paying the majority wait once per batch and reaching about 0.22 milliseconds per record and 4,600 audits per second on a three-node replica set.

Scaling the audit write path

Each run fans out 100k events × ~100 rules ≈ 10M audit writes. In V0 every write blocked the Kafka processing thread waiting for majority replication. The redesign moves persistence off that thread and batches it, then scales out across a cluster for parallelism and high availability.

What changed and why
Decoupled persistence
The Streams thread never touches Mongo — it hands audits to the audit-events topic, so DB latency can't stall evaluation.
Batch bulk writes
The audit-writer persists 500 records in one unordered bulkWrite, amortizing a single w:majority wait across the batch.
Replicated storage
A 3-node replica set replaces the single node for failover; idempotent _id upserts keep batch retries safe.
 Net effect: ~112ms → ~0.22ms per record (about 500× less write-concern wait). Throughput is now bounded by partitions and consumers, not by a blocking database call.
